## [KV Cache Quantization Cuts Compute Buffer 3x for DeepSeek-V4-Flash](https://www.reddit.com/r/LocalLLaMA/comments/1uk5y5m/deepseekv4flash_mxfp4_compute_buffer_scales_3x/) ⭐️ 8.0/10

A Reddit user discovered that switching the KV cache quantization from f16 to q8_0 reduces the compute buffer by approximately 3x for DeepSeek-V4-Flash in llama.cpp, from 12,964 MiB to 3,973 MiB at 10,240 context length. This optimization enables running DeepSeek-V4-Flash with much higher context lengths on consumer GPUs with 32 GB VRAM, which previously ran out of memory at 32,000 context with f16 cache. The compute buffer dropped from 12,964 MiB to 3,973 MiB (3.26x reduction) while the KV cache itself only shrank by ~200 MiB, indicating the compute buffer is the dominant memory consumer. The test used llama.cpp build 9851 with flash attention enabled and identical batch sizes.

reddit · r/LocalLLaMA · /u/Shoddy_Bed3240 · Jul 1, 00:01

**Background**: KV cache stores key-value pairs from previous tokens to avoid recomputation during autoregressive decoding. Quantizing this cache from f16 (16-bit float) to q8_0 (8-bit integer) reduces memory usage, but the compute buffer—a separate memory region for intermediate computations—can be much larger. For DeepSeek-V4-Flash, which uses a compressed KV cache (CSA/HCA/lightning-indexer), the compute buffer dominates memory consumption.
