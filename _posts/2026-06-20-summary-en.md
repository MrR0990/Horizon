## [Project Valhalla Arrives in JDK 28 After a Decade](https://www.jvm-weekly.com/p/project-valhalla-explained-how-a) ⭐️ 8.0/10

Project Valhalla introduces value types and heap flattening to the JVM, arriving in JDK 28 after a decade of development. This is a major JVM enhancement that improves performance and memory efficiency by allowing objects to be stored inline without headers or indirection, benefiting Java and other JVM languages. Value types are reference types without identity, meaning they cannot be null and are compared by value; heap flattening removes object headers and pointers, enabling dense storage in arrays.

hackernews · philonoist · Jun 19, 06:35 · [Discussion](https://news.ycombinator.com/item?id=48595511)

**Background**: Project Valhalla is a long-running effort to add value types to the JVM, similar to primitives but with methods and fields. Value types lack identity and are immutable, making them ideal for high-performance data structures. Heap flattening stores value objects inline in arrays and fields, reducing memory overhead and improving cache locality.

## [IETF Publishes RFC 10008 for New HTTP QUERY Method](https://www.ithome.com/0/966/439.htm) ⭐️ 8.0/10

The Internet Engineering Task Force (IETF) has published RFC 10008, defining the new HTTP QUERY method as a proposed standard. This method allows complex read-only requests with a request body, filling a gap between GET and POST. This standard provides a semantically correct way to perform complex queries without misusing POST, improving API design clarity and enabling safe, idempotent read operations. It will affect web developers, API designers, and all HTTP-based services. QUERY is defined as safe and idempotent, similar to GET, but allows a request body like POST. The specification also introduces the Accept-Query response header and supports caching with URI assignment for subsequent GET requests.

rss · IT之家 · Jun 19, 15:16

**Background**: HTTP methods like GET and POST have distinct semantics: GET is for safe, idempotent data retrieval but has URL length limits, while POST can carry a body but is not guaranteed to be read-only. Developers often used POST for complex queries, which was semantically incorrect. The QUERY method resolves this by combining the safety of GET with the body-carrying capability of POST.
